Component Notes

chatbot-service

  • Owns API endpoints, session state, message persistence, and idempotency.
  • Runs deterministic safety before any agent output is trusted.
  • Controls handoff to operators and final response delivery to the user channel.
  • Good fit as the stable application gateway on both Kind and hosted Kubernetes.

agent-service

  • Groups all agents into one deployable AI service instead of one pod per agent.
  • Owns prompts, model adapters, fallback logic, and structured reasoning outputs.
  • Can scale independently from chatbot-service as AI traffic grows.
  • Keeps the architecture clean without over-fragmenting the cluster.

Supporting Services

  • graph-core remains the domain intelligence and retrieval engine.
  • resource-svc handles shelters, clinics, NGOs, and other structured resource lookup.
  • geo-svc handles geocoding, reverse geocoding, routing, and ETA.
  • comms-svc handles outbound channels and can later absorb transcription adapters.

Data Layer

  • PostgreSQL + pgvector stores sessions, graph records, cases, embeddings, and audit data.
  • MinIO stores media and attachments for current and future multimodal workflows.
  • Model access is shown separately so the AI dependency stays isolated to the agent layer.
  • This layout works well on Kind first and maps cleanly to a hosted cluster later.
