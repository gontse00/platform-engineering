Agent Details

Intake Agent

  • Handles the primary conversation with survivors
  • Extracts structured fields through empathetic dialogue
  • Calls Resource Directory MCP to find nearby help
  • Calls Case Management MCP to create/update cases
  • Calls Geolocation MCP for address resolution
  • Uses GPT-4o-mini with function calling
  • Falls back to keyword extraction if LLM unavailable

Triage Agent

  • Assesses urgency and safety risk from context
  • Considers compound risk factors (DV + injury + no shelter)
  • Calls Crisis Knowledge MCP for protocol lookup
  • Output is bounded by deterministic safety floor
  • Can raise urgency but never lower it below keyword minimum
  • Triggers auto-escalation for critical/urgent cases

Safety Plan Agent

  • Generates personalized safety plans post-triage
  • Calls Geolocation MCP for escape routes
  • Calls Resource Directory MCP for nearest shelters
  • Calls Crisis Knowledge MCP for DV safety checklists
  • Stores plan as a graph node linked to the case
  • Triggered for DV, assault, and trafficking cases

Follow-up Agent

  • Runs on a schedule (e.g., 24h after case creation)
  • Checks case status via Case Management MCP
  • Sends check-in messages via Comms MCP
  • "Are you safe?", "Did you reach the shelter?"
  • Can re-escalate if survivor reports new danger
  • Closes cases after confirmed resolution

Translation Agent

  • Detects language via Comms MCP
  • Translates survivor messages to English for processing
  • Translates bot responses back to survivor's language
  • Supports all 11 SA official languages
  • Runs as a preprocessing/postprocessing layer
  • Preserves original text for legal records

MCP Design Principles

  • Each MCP server is a standalone process
  • Agents discover tools dynamically via MCP protocol
  • MCPs are stateless — all state lives in the data layer
  • Any MCP can be used by any agent (shared toolbox)
  • MCPs can also be used from Kiro/Claude Desktop for debugging
  • Deployed as pods in Kind or run locally with uvx
